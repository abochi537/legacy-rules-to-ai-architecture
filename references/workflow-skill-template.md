# Workflow Skill Template

Create workflow skills only after a pattern has repeated or the user asks to
make a reusable procedure. Store them where the user prefers: global Codex
skills, Hermes skills, a shared team repo, or project-local skills.

## When To Create A Workflow Skill

- The same migration/development workflow appears in multiple features.
- The workflow has enough project-specific nuance that generic AI guidance
  causes drift.
- The workflow references stable docs in `<ai-docs-dir>/*`.
- The skill can be verified with a realistic prompt.

## Boundary

- Skills store procedures.
- `<ai-docs-dir>/*` stores project facts.
- `AGENTS.md` stores the entry contract and hard rules.

Do not copy the whole domain rule set into a skill. Link to the relevant
project context file and tell the future agent which section to read.

## SKILL.md Pattern

```md
---
name: <verb-led-name>
description: <one sentence describing the workflow and trigger context>
---

# <Human Title>

Use this skill when <specific trigger>. It assumes the target project has an
AI context directory generated by `legacy-rules-to-ai-architecture`.

## Before Acting

1. Read `AGENTS.md`.
2. Read `<ai-docs-dir>/00-index.md`.
3. Read only the task-relevant context files:
   - Frontend: `frontend-guidelines.md`, `architecture.md`, `domain-rules.md`
   - API/backend: `api-contracts.md`, `architecture.md`, `testing-strategy.md`
   - Migration: `migration-map.md`, `domain-rules.md`, `testing-strategy.md`

## Procedure

1. Identify the target feature/page/API and existing canonical pattern.
2. Locate evidence before implementing.
3. Reuse approved architecture and UI patterns.
4. Update context docs if a rule changes or an unknown becomes known.
5. Run required checks from `AGENTS.md`.

## Verification

- Report files changed.
- Report rules consulted.
- Report checks run and any skipped checks.
```

## Suggested Workflow Skills

| Skill | Purpose |
| --- | --- |
| `migrate-feature` | Move one legacy feature into the target architecture with characterization tests. |
| `add-api-endpoint` | Add or change an API while preserving contract docs and tests. |
| `add-page` | Add a frontend page using canonical layouts, components, and visual checks. |
| `preserve-frontend-consistency` | Review UI changes for drift against page patterns and screenshots. |

## Validation

Before sharing a generated workflow skill, test it on one realistic prompt and
compare the result against the same prompt without the skill. Keep the skill if
it reduces drift, missing context, or repeated prompting.
